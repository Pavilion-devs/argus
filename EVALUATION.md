# Argus вЂ” Evaluation, Benchmark & Engineering Rigor

Argus's core claim is that *every conclusion is provable from the data it queried.* That
claim is only credible if the agent itself is **measured, not asserted** вЂ” and measured
honestly. This document is the evidence: how we evaluate, the curated benchmark, the
results, and вЂ” most importantly вЂ” the **bugs the harness caught and how we root-caused
them.** A benchmark that only ever shows green is not measuring anything; ours earned its
keep by repeatedly catching real problems, including in its own ground truth.

---

## 1. The evaluation harness (`argus eval`)

Argus is scored against curated **Boss of the SOC (BOTS) v3** scenarios with known ground
truth. For each scenario the harness runs a full, real, autonomous investigation against
live Splunk (via MCP) and measures:

| Metric | What it proves |
|---|---|
| **Verdict accuracy** | Does Argus reach the right true/false-positive call? (incl. benign controls so it doesn't cry wolf) |
| **Indicator recall** | Did it surface the key ground-truth IOCs? |
| **Grounding precision** | Does **every** concrete IOC it reports actually exist in the data? (an automated zero-hallucination check) |
| **ATT&CK validity** | How many reported MITRE technique IDs are invalid? (validated against the pinned ATT&CK v19.1 catalog вЂ” target: 0) |
| **Efficiency** | Queries run + wall-clock per investigation |

Two design choices make the numbers trustworthy:

- **Multi-sampling (`--repeat K`).** A single run of a non-deterministic agent is one
  noisy draw. `--repeat K` runs each scenario K times and reports a **verdict pass-rate +
  distribution**, so a borderline scenario shows up as `2/3`, not a misleading `pass`/`fail`.
- **Data-verified ground truth.** Every expected indicator is confirmed to exist in
  `botsv3` *and* is curated as verified-**malicious** (see Bug #4 for why "exists" is not
  enough). Ground truth is documented inline in `eval.py` with a malice rationale.

## 2. The benchmark вЂ” 6 scenarios (4 attacks + 2 precision controls)

Every indicator below was confirmed in the BOTS v3 data before being used as ground truth.

| Scenario | Type | Verdict | Key indicators | Data-verified basis |
|---|---|---|---|---|
| `aws_cred_abuse` | IAM credential abuse | true_positive | `web_admin`, `139.198.18.205` | Compromised IAM user driving 637 denied RunInstances (cryptomining) from a China IP |
| `endpoint_malware` | Endpoint malware | true_positive | `iexeplorer.exe` | Masquerading binary (typosquat) from a temp path; Sysmon EventID 1 shows a reverse shell + Struts exploit + `/etc/passwd` read |
| `s3_public_exposure` | Cloud data exposure | true_positive | `frothlywebcode`, `bstoll` | `PutBucketAcl` granting the **AllUsers** group public READ on a code bucket |
| `coinminer_cryptojacking` | Cryptojacking | true_positive | `coinhive.com` | Repeated DNS lookups to the Coinhive Monero-mining service |
| `benign_dns_control` | **Precision control** | false_positive / inconclusive | вЂ” | DNS to `8.8.8.8` is `chrome.exe` resolving `google.com` via Google's public resolver |
| `benign_aws_recon_control` | **Precision control** | false_positive / inconclusive | вЂ” | "Recon-looking" API enumeration is `splunk_access`, the Splunk Add-on for AWS's own data collector |

The two benign controls are deliberate: a SOC agent that flags real attacks but also cries
wolf on its own monitoring is useless. They test **precision**, and they have caught the
agent over-calling more than once (Bugs #3 and #5).

## 4. Engineering rigor вЂ” bugs we caught and root-caused

The discipline throughout: **treat a failing metric as a clue, not the target.** Several of
these would have been easy to "fix" at the symptom level in a way that made the system
*worse*; root-causing them is what made it better.

### Bug #1 вЂ” Grounding precision was measuring almost nothing
**Symptom:** the flagship `aws_cred_abuse` scenario reported `grounding_precision: null`
(0 IOCs checked) despite the agent surfacing 18 indicators.
**Root cause:** the check scored only the report's `iocs` field, but the model often files
indicators under `affected_entities`. The headline zero-hallucination metric was silently
checking nothing.
**Fix:** ground the full concrete IOC set (iocs + entity values). Grounding precision became
a real measurement (now ~0.99 over the benchmark).

### Bug #2 вЂ” Endpoint malware verdict under-calling
**Symptom:** the masquerading-executable scenario flipped `true_positive` / `inconclusive`
~50/50; the hedged runs sat at confidence в‰€ 0.45.
**Root cause:** Sysmon data in `botsv3` is raw XML and **not field-extracted**, so
`EventCode=`/`Image=` filters returned nothing вЂ” the agent couldn't reach the decisive
EventID 1 lineage (which contains the reverse shell + Struts exploit) and gave up.
**Fix:** (a) taught the agent the Sysmon-XML extraction tradecraft (`rex` on `_raw`), and
(b) added a **confidence-gated continuation** вЂ” if the first synthesis hedges to a
low-confidence `inconclusive`, run one more focused pass before finalizing. Verified 3/3
true_positive; observed live going `inconclusive @0.45 в†’ continuation в†’ true_positive @0.92`.

### Bug #3 вЂ” The fix for #2 caused a precision regression
**Symptom:** after Bug #2's first fix, the benign DNS control started returning
`true_positive` (1/3) вЂ” over-calling.
**Root cause:** the "don't under-call" guidance was **asymmetric** вЂ” it pushed toward
finding threats, so on a benign alert the agent over-pivoted into unrelated host activity.
**Fix:** made the verdict logic **symmetric and alert-scoped** вЂ” confirming activity is
benign is an equally valid, confident outcome, and unrelated/ambient activity must not be
promoted into a verdict for *this* alert. Benign control returned to 3/3 with endpoint
holding at 3/3. (Catching this is exactly why precision controls exist.)

### Bug #4 вЂ” Poisoned ground truth (the benchmark was wrong, not the agent)
**Symptom:** `aws_cred_abuse` recall stuck at 0.5 вЂ” the agent never surfaced `34.215.24.225`.
**Root cause (traced through the pipeline):** in the data, `34.215.24.225` is
`arn:aws:iam::622676721278:user/splunk_access` вЂ” the **Splunk Add-on for AWS's own
data-collection account** (userAgent `Boto3/вЂ¦ Linux-aws`, 4040 benign read-only calls, zero
malicious actions, never touches `web_admin`). It had been wrongly listed as an "attacker
IP," so the harness was **penalizing the agent for correctly refusing to flag benign
infrastructure.** Deeper systemic cause: ground truth was validated for *existence*, not
*malice*.
**Fix:** corrected the ground truth to the real, data-verified core IOCs (`web_admin`, the
compromised account, and `139.198.18.205`, the primary attacker IP). **Recall went
0.58 в†’ 1.0 with no change to the agent** вЂ” proof it was a benchmark bug. Had we "fixed the
symptom" by nudging the agent to surface `34.215.24.225`, we would have *trained the exact
over-calling Bug #3 had just removed.*

### Bug #5 вЂ” Over-calling benign API enumeration as "recon"
**Symptom:** a new precision control (`benign_aws_recon_control`, built from the Bug #4
finding) caught the agent calling the benign `splunk_access` enumeration `true_positive`
~1/2 the time.
**Root cause:** the agent confirmed "enumeration = recon" without identifying the
**principal** behind it.
**Fix:** general tradecraft вЂ” before judging enumeration/scanning, identify the account +
userAgent and check for *denied/abusive* actions; a service account with an SDK/CLI
userAgent doing read-only calls with no denials is benign automation, not an attacker. Now
3/3 benign, ~40% fewer queries, and real recon (compromised identity + denied actions)
still flags correctly.

## 5. The discipline (what we will not do)

- **No gaming the metric.** We do not re-roll the eval until a lucky 100% appears, revert
  settings just to pass, relax a verdict to make a hard scenario pass, or name a test's
  ground-truth answer in any prompt (examples in prompts are deliberately generic).
- **Data-verified ground truth only.** An indicator must be confirmed malicious in the
  data, not merely present (Bug #4).
- **Root cause over symptom.** A bad metric is investigated to its origin вЂ” which is
  sometimes the agent, sometimes the prompt, and sometimes the benchmark itself.
- **Honesty in the open.** Known limitations live in [`issues.md`](issues.md); resolved
  problems and their evidence live in [`problems.md`](problems.md).

---

## 6. Known limitations (honest)

- Indicator recall is the softest metric (~0.92): IOC *enumeration completeness* varies
  run-to-run even when the verdict and grounding are correct.
- 6 scenarios is a focused benchmark, not exhaustive coverage of the attack surface;
  breadth (lateral movement, exfiltration, more SaaS scenarios) is the next rigor step.
- Multi-sampling at `--repeat 3` is a strong signal, not a guarantee; higher `K` tightens
  confidence intervals.
